# Contracts — Sepolia milestones 1 and 3

Milestone 1 was leanVM's hash on-chain (below). Milestone 3 adds the four
contracts and ports `pool/envelope.py`'s attack battery to Foundry.

## The contracts (milestone 3)

```
src/NonceManager.sol     EIP-8250 emulation: per-sender keyed-nonce domains
                         (slot(sender, key) = keccak256(pad(sender) || key)),
                         consume-fresh-at-0. The per-sender part is why the
                         pool pins one sender.
src/RecentRoots.sol      EIP-8272 emulation: (source_id, slot, root) entries,
                         8191-slot window, block.number standing in for the
                         consensus slot.
src/AttestedVerifier.sol the trust shim: an attester signs (claim, proofHash);
                         this checks the signature. It is exactly the boundary a
                         real devnet's leanVM-verify precompile deletes. Anyone
                         re-verifies a posted proof with prover/verify-spend.
src/ShieldedPool.sol     the immutable pool: incremental Merkle tree over
                         Poseidon16, shield/transfer/withdraw, and the four
                         VERIFY bindings from devnet/README.md (pinned sender,
                         key set, root reference, operation shape) plus the
                         duplicate-append no-op.
test/ShieldedPool.t.sol  the envelope.py battery on-chain (cross-sender
                         double-spend, lifted/unattested proof, foreign and
                         stale roots, transfer+withdraw mint, duplicate no-op,
                         withdraw payout) + cross-stack fixtures: ctxFor,
                         computeClaim, and the incremental tree root all checked
                         against values the leanVM prover / Python reference
                         produced (vectors/pool_fixtures.json).
```

`forge test` runs all of it (23 tests). The leanVM STARK is represented in these
tests by the attester signature; what is tested is the pool's binding logic,
not the proof. The real proof is checked end to end by
[`../prover/`](../prover/) (milestone 2) and wired in at milestone 4.

To regenerate `vectors/pool_fixtures.json` (after any hash or encoding change):

```
cd ../prover && cargo build --release --bin prove-spend
./target/release/prove-spend --demo 20 --recipient 0x00000000000000000000000000000000cafebabe \
    --out /tmp/wd_pub.json --proof-out /tmp/wd_proof.json
python3 ../contracts/reference/gen_pool_fixtures.py   # reads /tmp/wd_pub.json
```

# Contracts — Sepolia milestone 1: the hash, on-chain

First milestone of the Sepolia path: leanVM's hash reimplemented in Solidity
and differentially tested against vectors exported from the exact leanVM
commit the spend circuit is proved on (`12e6151`). Everything downstream (the
pool contract's claim recompute, the on-chain Merkle tree) trusts this
library, so it ships with its own reference chain: Rust (leanVM itself) →
JSON vectors → Python (`reference/poseidon16.py`) → Solidity
(`src/Poseidon16.sol`), all agreeing bit for bit.

One naming correction this work surfaced: leanVM's `poseidon16_compress` is
**classic Poseidon, not Poseidon2** (upstream calls it `Poseidon1KoalaBear16`):
width 16 over KoalaBear (p = 2^31 - 2^24 + 1), x^3 S-box, 4 + 20 + 4 rounds,
circulant MDS with first column [1, 3, 13, 22, 67, 2, 15, 63, 101, 1, 2, 17,
11, 1, 51, 1], and compression = permute + feed-forward, truncated to 8 lanes.
Repo docs previously said "Poseidon2"; they have been corrected.

## Layout

```
src/Poseidon16.sol        the library: permute, compress, compressPair, tagged,
                          packDigest/unpackDigest (the canonical encoding from
                          devnet/README.md). GENERATED by reference/gen_poseidon_sol.py.
test/Poseidon16Vectors.t.sol  differential tests: every exported vector, the pool's
                          owner_pk/cm/nf/claim chain, canonical-encoding rejection,
                          flip-a-lane soundness smoke, and a gas-regression ceiling.
vectors/poseidon16_vectors.json  64 permute/compress vectors + the pool chain,
                          exported from leanVM 12e6151.
reference/poseidon16.py   plain-Python reference (validates the same vectors;
                          the wallet-side building block).
reference/round_constants.json   28 x 16 round constants (transcribed from leanVM).
reference/gen_poseidon_sol.py    generates src/Poseidon16.sol (Yul core, unrolled MDS).
(vectors regenerate via ../prover: cargo run --release --bin export-vectors).
```

## Run

```
# Solidity vs leanVM vectors (needs Foundry):

# Python vs the same vectors:
```

## Measured

One `compress` (the Merkle/tagged two-to-one primitive) costs **~164k gas**
(via-ir, optimizer 200 runs; the test suite enforces a 200k ceiling). That
prices the pool's on-chain steps at roughly: claim recompute (4 compresses)
~0.7M gas, incremental tree append ~164k per level (~3.3M at depth 20). Usable
for a Sepolia demo, with clear optimization headroom (fewer memory round
trips, batched reductions, or the FFT-based MDS leanVM itself uses) if it ever
matters; correctness parity was this milestone's goal.
